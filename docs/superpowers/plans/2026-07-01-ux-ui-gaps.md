# UX/UI Gap Fixes Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the UX/UI gaps introduced when the ERP dashboard moved from single-user localStorage to multi-user realtime Firestore — surface save failures, add a loading state, add logout, and make destructive shared-reset actions safe.

**Architecture:** All changes stay in the persistence/login/settings layer of the single-file app (`index.html`) plus the two ES modules (`js/auth.js`, `js/db.js`). No dashboard render/HTML/CSS behaviour changes except the explicitly-requested new UX affordances (loading overlay, logout, sync badge). Additive HTML/CSS + small edits to existing persistence helpers.

**Tech Stack:** Single-file HTML app (classic `<script>` with global functions), Firebase Web SDK v11.10.0 modular (ES modules `js/db.js` + `js/auth.js` bridged via `window.DB`), Firestore realtime, GitHub Pages hosting.

## Global Constraints

- Dashboard UI/rendering must stay **identical** — do not touch `renderAll`, `renderPI`, `renderBridge`, `renderPlan`, `apSegs`, `openPI`, `openAP`, drawer, drag/zoom, layout, colors, or button behaviour of the existing app. Only edit: persistence helpers, `js/auth.js`, `js/db.js`, the login gate, the Settings panel, and add clearly-additive overlays/badges.
- All DATA persistence stays per-item on Firestore; per-user prefs (`erp_theme`, `erp_accent`) stay in localStorage. Never reintroduce localStorage for DATA.
- Firebase SDK pinned to **v11.10.0** (gstatic modular URLs) — do not change the version.
- Keep existing element ids used by `js/auth.js`: `loginForm`, `loginPass`, `loginBtn`, `loginErr`, and the `busy` class on `loginBtn`.
- **Verification cycle per task** (this project has no unit-test framework):
  1. `node --check` the extracted main script (command in the Verify Helper below).
  2. Playwright smoke: serve locally, bypass the gate via the graceful fallback path, confirm no render regression + the new feature is present.
  3. Commit; push only at the end of a phase (Pages auto-rebuilds ~1-2 min).
- Deploy = `git add -A && git commit && git push origin main`. `firebase-config.js` is committed (non-secret, rules locked to team email); `serviceAccountKey.json` stays gitignored.

### Task 5: Connection / sync status badge

- Modify: `index.html` — add a small `#syncBadge` (fixed, bottom-left) subscribed to status.

**Interfaces:**
- Produces: `window.DB.onStatus(cb: (s:{online:boolean, saving:boolean})=>void): void` and increments a pending counter around each write.

- [ ] **Step 1: In `js/db.js`, add status tracking.** Add near the adapter:

```js
let _pending=0, _online=true, _statusCb=null;
function _emit(){ if(_statusCb) _statusCb({online:_online, saving:_pending>0}); }
window.addEventListener('online', ()=>{_online=true;_emit();});
window.addEventListener('offline', ()=>{_online=false;_emit();});
```

Wrap `saveItem`/`updateItem`/`deleteItem` bodies with `_pending++; try{ await ... } finally { _pending--; _emit(); }` and call `_emit()` once at start. Add to the `DB` object: `onStatus(cb){ _statusCb=cb; _emit(); }`.
